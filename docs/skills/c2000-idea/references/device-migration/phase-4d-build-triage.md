# Phase 4D — Build Error Triage

> You are a **sub-agent** executing Phase 4D. You were dispatched by the Phase 4
> orchestrator after Phase 4C (sweep) returned a build that still has errors.
>
> Your job is to classify each build error, apply the correct fix strategy, rebuild,
> and repeat until the build is clean — or until you hit the convergence guard limit.
>
> **Do not** modify any file that is not directly named in a build error. Do not
> invent symbol names or API signatures from memory — query the IDEA MCP for every
> replacement. Record every triage action in `c2000-migration.md`.

---

## When Phase 4D is invoked

Phase 4D is dispatched only when Phase 4C reports a **non-passing build**
(i.e., `buildProject()` returned errors after Phase 4C's second clean build attempt).

## Briefing fields — confirm before starting

The orchestrator briefing must include all fields in the table below. Before taking any
action, read the briefing and confirm every field is present. If a field is missing or
unclear, **stop and ask the orchestrator (or the user) to supply it** — do not assume
or invent values.

| Field | What it is | Where it comes from |
|---|---|---|
| `targetProjectName` | CCS project name of the target project | Phase 1 (renameProject result) |
| `targetProjectDir` | Absolute path to the target project directory on disk | Phase 1 (getProjectDescriptors) |
| `sourceDevice` | Source device family name (e.g., `f28003x`) | Phase 1 (IDEA MCP get_projects) |
| `targetDevice` | Target device family name (e.g., `f28p55x`) | Phase 1 (IDEA MCP get_projects) |
| `c2000ware_path` | Absolute path to the C2000Ware root | Phase 1 (SDK resolution) |
| `activeBuildConfig` | Active build configuration name (e.g., `CPU1_FLASH`) | Phase 2 (getToolFlags) |
| `sysConfigOutputLocation` | SysConfig-generated output folder path | Phase 3 (getProjectDescriptors) |
| `buildErrorOutput` | Full verbatim error output from Phase 4C's final buildProject call | Phase 4C structured result |
| `migrationLogSummary` | Phase 4 progress table and deferred-errors list from c2000-migration.md | Phase 4C structured result |

> **`sysConfigOutputLocation` — read-only guard:**
> Files under `sysConfigOutputLocation` are **auto-generated by SysConfig** every time
> SysConfig runs. Any manual edits to these files will be silently overwritten.
> **Do not edit any file whose path starts with `sysConfigOutputLocation`.**
> If a build error points to a file under this path, classify it as `DEFERRED-MANUAL`
> with note: `Category X: error in SysConfig-generated file — fix in SysConfig (.syscfg),
> not in the generated source`.

---

The orchestrator provides you with:
- Target project name
- Source and target device names
- The build error output from Phase 4C's final `buildProject()` call
- The current `c2000-migration.md` contents (for context)

---

## Convergence guard

**Maximum iterations: 3.**

Each iteration = one full (classify → fix → rebuild) cycle.

If after 3 iterations the build is still not clean:
- Record `Phase 4D: CONVERGENCE LIMIT REACHED` in `c2000-migration.md`.
- List all remaining errors as `DEFERRED-MANUAL` items.
- Return control to the orchestrator with status `⚠ BUILD NOT CLEAN — manual items remain`.
- **Do not continue past 3 iterations.** An unconverging triage loop is a signal that
  the errors require human judgement or upstream investigation.

---

## Step 4D.1 — Collect the current build error list

Call `buildProject(<target project name>)` to get the **current** error output.

> Even if the orchestrator provided a prior error list, call `buildProject()` fresh —
> Phase 4C may have partially resolved some errors that are no longer present.

> **⚠ MCP hang guard — `buildProject` may not respond:**
> `buildProject` is a long-running synchronous call. If the MCP tool call has produced
> **no response at all** (no result, no error, no progress output) after a long wait
> (typically 2–3 minutes), assume the tool has hung. Do **not** keep waiting.
> Immediately:
> 1. Record in `c2000-migration.md`:
>    `HANG: buildProject(<project>) — no response after timeout. Phase 4D, step 4D.1, iteration <N>.`
> 2. Stop and tell the user:
>    *"The `buildProject` call has not responded. The MCP tool may have hung.
>    Please check the CCS console, restart the MCP server if needed, and tell me
>    whether the build succeeded or failed so I can continue."*
> 3. Wait for the user's response before taking any further action.

Parse the output into individual error entries. For each entry, note:
- File path and line number
- Error message text
- Error code (if provided by the compiler)

Record the error count:
```
4D iteration <N>: <X> errors
```
in `c2000-migration.md`.

If `buildProject()` returns **0 errors** → the build is already clean. Skip to Step 4D.5.

---

## Step 4D.2 — Classify each error

For each error, assign it to one of the four categories below. A single error maps to
exactly one category — use the first matching rule.

### Category A — Missing include / file not found
**Pattern:** `fatal error: <header>.h: No such file or directory`

**Fix strategy:**
1. Check whether the missing header name contains the **source device name** (e.g.,
   `f28002x_` prefix). If so, this is a Phase 4A/4B include-path miss.
2. Call `get_device_migration_report(<affected file path>, <source device>, [<target device>])`
   on the file containing the `#include` to get a collateral link or suggested fix for
   the missing header.
3. If the correct target-device header is identified, update the `#include` directive
   in the affected file.
4. If no replacement is found, mark as `DEFERRED-MANUAL` with note:
   `Category A: no replacement found for <header>.h`

### Category B — Undefined symbol (function or variable)
**Pattern:** `undefined reference to '<symbol>'` or `error: '<symbol>' undeclared`

**Fix strategy:**
1. Call `get_device_migration_report(<file path>, <source device>, [<target device>])`
   on the file containing the error to get the authoritative replacement.
2. If the report provides a `Suggested fix` → apply it directly.
3. If the report provides no `Suggested fix` → call the TI ASM MCP to look up the
   symbol in the target device's TRM (register definition or peripheral chapter).
4. Apply the replacement only if you have a confirmed authoritative source (IDEA MCP
   or TI ASM MCP). **Do not invent a replacement from memory.**
5. If no authoritative replacement is found → mark as `DEFERRED-MANUAL` with note:
   `Category B: no authoritative replacement for '<symbol>'`

### Category C — Deprecated or renamed API
**Pattern:** `error: implicit declaration of function '<api>'` or
`warning-as-error: '<api>' is deprecated`

**Fix strategy:**
1. Call `get_device_migration_report(<file path>, <source device>, [<target device>])`
   on the affected file.
2. Identify the entry for `<api>` in the report output.
3. Apply the `Suggested fix` from the report.
4. If no entry exists in the report → query TI ASM MCP for the target device peripheral
   chapter to confirm whether the peripheral itself still exists on the target.
5. If the peripheral does not exist on target → mark as `DEFERRED-MANUAL` with note:
   `Category C: peripheral absent on target — <api>`

### Category D — Linker / memory map error
**Pattern:** `error: region '<region>' overflowed by <N> bytes` or
`undefined symbol: <section>` in linker output

**Fix strategy:**
1. This is out of scope for automated symbol replacement. Do **not** modify linker
   scripts or `.cmd` files here — linker map changes require user decision.
2. Immediately mark as `DEFERRED-MANUAL` with note:
   `Category D: linker region error — <region> — requires manual .cmd edit`
3. Tell the user:
   > *"Linker region error detected in `<region>`. This requires manual review of the
   > `.cmd` linker script for the target device. I've recorded it as DEFERRED-MANUAL."*

---

## Step 4D.3 — Apply fixes

For each error classified as A, B, or C (not D, not `DEFERRED-MANUAL`):

1. Open the affected file.
2. Apply the replacement identified in Step 4D.2.
3. For each fix, record in `c2000-migration.md`:
   ```
   4D fix: <file>:<line> — <category> — replaced '<old>' with '<new>'
   ```
4. Do **not** make speculative changes (e.g., "this might also be wrong"). Fix only
   what the build error directly identifies.
5. After applying all fixes for this iteration, proceed to Step 4D.4.

---

## Step 4D.4 — Rebuild and evaluate

Call `buildProject(<target project name>)` again.

> **⚠ MCP hang guard — `buildProject` may not respond:**
> Same rule as Step 4D.1. If `buildProject` produces no response after a long wait
> (2–3 minutes), record in `c2000-migration.md`:
> `HANG: buildProject(<project>) — no response after timeout. Phase 4D, step 4D.4, iteration <N>.`
> Stop and ask the user whether the build succeeded or failed before continuing.

- **0 errors** → build is clean. Proceed to Step 4D.5.
- **Same errors as before (no reduction)** → the fixes had no effect. This counts as
  a non-productive iteration. Increment the iteration counter. If at limit → invoke
  the convergence guard (top of this file). Otherwise, re-classify the same errors
  more carefully — check whether the file paths are correct and that the edit was
  actually saved.
- **Fewer errors than before** → progress is being made. Increment iteration counter.
  If at limit → proceed to convergence guard. Otherwise, loop back to Step 4D.1 with
  the remaining errors.
- **New errors not present in prior iteration** → fixes introduced a regression. Do
  not apply further speculative fixes. Record the new errors as `DEFERRED-MANUAL` and
  proceed to Step 4D.5 with the regression noted.

---

## Step 4D.5 — Record results and return to orchestrator

Update `c2000-migration.md`:

```
## Phase 4D — Build Triage

Iterations used: <N> / 3
Errors at start: <X>
Errors resolved: <Y>
Errors remaining: <Z>
DEFERRED-MANUAL items: <list each>

Phase 4D status: <CLEAN BUILD | CONVERGENCE LIMIT | REGRESSION DETECTED>
```

Then update the Phase status table:
```
| Phase 4D — Build Triage | <✅ COMPLETE | ⚠ MANUAL ITEMS> | <summary> |
```

Return the following structured result to the orchestrator:

```
Phase 4D result:
Status: <CLEAN BUILD | BUILD NOT CLEAN — manual items remain>
Iterations: <N>
Fixed: <Y> errors
Remaining: <Z> errors
DEFERRED-MANUAL: <list, or "none">
Final build: <PASS | FAIL>
```

The orchestrator will update the overall Phase 4 summary and decide whether to
proceed to Phase 5 or present the remaining items to the user.
